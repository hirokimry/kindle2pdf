## ⚖️ 0. 前提と法的スタンス（重要）

本システムは **DRMを解除しない**。Kindleアプリに画面表示させた内容をスクリーンショットで取得し、OCRを付与してPDF化する、いわゆる「自炊」に近い手法を採る。

- Kindle本のDRM（技術的保護手段）を回避して`.azw`等を直接変換する方式は、日本の著作権法上「技術的保護手段の回避」に該当し、私的複製の例外から外れる。本システムは**その方式を採用しない**。
- スクショ方式はDRMそのものを解除しないため相対的に穏当だが、**利用はあくまで自分が購入した書籍の個人的利用の範囲に限定する**。再配布・共有は行わない。
- 生成物には元書籍の著作権が及ぶ。取り扱いは自己責任の前提とする。

## 🎯 1. 要件定義

### 1.1 目的

購入済みKindle書籍を、**Mac上で全自動**でページ送り・撮影・OCRし、**文字検索・コピーが可能なPDF**（画像＋透明テキスト層）として保存する。

### 1.2 対象書籍

| 種別 | 対応方針 |
|------|----------|
| リフロー型（小説・実用書などテキスト主体） | **主対象**。Kindleのフォントサイズ・ウィンドウを固定してページを安定化 |
| 固定レイアウト型（コミック・雑誌・技術書の一部） | 対応可。OCR精度は画質依存。テキスト層は付くが図表内文字は取りこぼしあり |

### 1.3 機能要件

- **F-1** Kindleアプリを対象書籍が開かれた状態から、1ページ目〜最終ページまで自動でページ送りする。
- **F-2** 各ページを指定領域でスクリーンショット取得する。
- **F-3** 最終ページを自動検出して撮影を停止する。
- **F-4** 取得画像から重複・遷移途中フレームを除去する。
- **F-5** 各ページ画像を余白トリミング・DPI正規化する（任意）。
- **F-6** Apple Vision（OCR）で日本語テキストと座標を抽出する。
- **F-7** ページ画像の上に透明テキスト層を重ね、**検索可能な単一PDF**に結合する。
- **F-8** 中断／再開（レジューム）に対応する。

### 1.4 非機能要件

| 項目 | 目標 |
|------|------|
| 画質 | 本文が判読可能かつOCR成立するDPI（実測で調整、目安2x Retina相当） |
| 速度 | 1ページあたり撮影＋待機で概ね1〜3秒（描画待ち含む）。OCR/結合は後段バッチ |
| 容量 | 検索可能を維持しつつPNG→適切な圧縮でファイルサイズを抑制 |
| 再現性 | 設定ファイルで撮影範囲・待機時間・DPIを再現可能に |
| 安全性 | 個人利用限定・ローカル完結（外部送信なし） |

### 1.5 スコープ外（今回作らない）

- DRM解除、`.azw`/`.kfx`の直接変換
- 複数書籍のバッチ一括投入（まずは1冊単位。将来拡張）
- クラウド保存・OCR外部API利用

## 🧩 2. 技術選定

| 処理 | 採用 | 代替 | 選定理由 |
|------|------|------|----------|
| ページ送り（キー送出） | **AppleScript（System Events）** or **cliclick** | Quartz Event Services直叩き | 右矢印キー送出のみで足りる。cliclickはbrew導入で確実。Accessibility権限必要 |
| スクリーンショット | **`screencapture`（標準）** `-R x,y,w,h`で領域指定 / `-l windowid`で窓指定 | Quartz `CGWindowListCreateImage` | 標準コマンドで領域撮影が安定。UI/余白を除いた読書領域だけを撮れる |
| 最終ページ検出 | **画像ハッシュ差分**（連続Nフレーム同一で停止） | ページ番号OCR監視 | アプリ外からページ総数を取れないため、遷移が止まったことを画像で検出 |
| 重複/遷移フレーム除去 | **perceptual hash（pHash）** | 画素差分（MSE） | 描画途中フレームや連打の重複を吸収 |
| OCR | **Apple Vision（`ocrmac`）** `language_preference=['ja-JP','en-US']` | Tesseract（`jpn`） | 日本語精度が高く座標付き。macOS標準機能で追加辞書不要 |
| PDF結合＋透明テキスト層 | **`img2pdf`＋`reportlab`**（Vision座標で不可視テキストを重ねる） | `ocrmypdf`（Tesseract経由） | Visionの高精度結果を活かし、画像はロスレスで保持しつつ検索層を付与 |
| 全体制御 | **Python 3**（+ `uv`/venv） | シェルスクリプト | 画像処理・OCR・PDF・状態管理を一元化 |

> **OCR経路の判断**: 最短で「検索可能PDF」を作るだけなら `ocrmypdf --language jpn` が手軽だが、日本語精度はVisionが優位。本設計は **Vision（ocrmac）で座標付き抽出 → reportlabで透明テキスト層** を主経路とし、`ocrmypdf`を簡易フォールバックとして残す。

## 🏗 3. アーキテクチャ・処理フロー

### 3.1 全体パイプライン

```
[準備] Kindleアプリで対象書籍を開く / 1ページ目 / 単一ページ表示 / フォント固定
   │
   ▼
① キャプチャ段（capture）        ← リアルタイム制御（Kindle操作）
   ・領域スクショ取得
   ・右矢印でページ送り
   ・pHashで「進んだ/止まった」を判定
   ・最終ページ検出で停止
   │  raw/ に連番PNGを出力（状態を state.json に逐次保存）
   ▼
② 前処理段（preprocess）         ← バッチ（Kindle不要）
   ・重複/遷移途中フレーム除去（pHash）
   ・余白トリミング / DPI正規化
   │  pages/ に確定ページPNG
   ▼
③ OCR段（ocr）                   ← バッチ
   ・Apple Vision で各ページ OCR（テキスト＋bbox）
   │  ocr/ に page_XXXX.json（text, bbox, confidence）
   ▼
④ PDF結合段（build）             ← バッチ
   ・各ページ画像を配置
   ・bbox座標に不可視テキストを重ねる
   │  output/<book>.pdf（検索可能）
```

### 3.2 モジュール分割（案）

- `capture.py` — Kindle制御・撮影・最終ページ検出（唯一Kindleに触る段）
- `preprocess.py` — 重複除去・トリミング・正規化
- `ocr.py` — Vision OCRラッパ（`ocrmac`）、JSON出力
- `build_pdf.py` — 画像＋透明テキスト層のPDF生成
- `pipeline.py` — 各段のオーケストレーション、レジューム制御
- `config.yaml` — 設定（撮影範囲・待機・DPI・閾値）
- `state.json` — 進捗（現在ページ・完了段・ハッシュ履歴）

### 3.3 段の分離方針

**キャプチャ段だけをリアルタイム（Kindle依存）**とし、②③④は撮影済み画像に対するバッチ処理として完全分離する。これにより、撮影失敗時の再実行やOCR/PDFのやり直しがKindle操作なしで可能になる。

## 🛑 4. 停止条件・エラーハンドリング（フル自動の肝）

### 4.1 最終ページ検出

- 右矢印送出 → 待機 → 撮影 → 直前ページとpHash比較。
- **連続 `N`回（既定3）同一ハッシュ**なら「これ以上進まない＝最終ページ」と判断し停止。
- 誤検出対策: 同一判定は「送り操作をしても画像が変化しない」場合のみ。読み込み遅延との区別のため、変化なし時は**追加待機してもう一度**撮る。

### 4.2 描画待ち・タイミング

- ページ送り後、`page_turn_wait`（既定0.8〜1.5秒）待ってから撮影。
- 「まだ描画中（ぼやけ/ローディング）」を避けるため、**連続2フレームが安定（ハッシュ一致）してから確定**するオプション。

### 4.3 異常系

| 事象 | 対処 |
|------|------|
| ページが進まない（末尾でない誤検出） | 待機延長＋リトライ`M`回。改善なければ停止して要確認ログ |
| Kindleウィンドウ非アクティブ/位置ズレ | 撮影前にウィンドウを前面化・座標再取得。ズレ検出時は中断 |
| スクショが真っ黒/想定外 | 明度チェックで異常フレームを検知しリトライ |
| 途中クラッシュ | `state.json`から**最終確定ページ以降を再開**（レジューム） |
| OCR失敗ページ | 当該ページのみ再OCR。失敗時は画像のみでPDF化（テキスト層なし）し、ログに記録 |

### 4.4 レジューム設計

- `state.json`に「段（capture/preprocess/ocr/build）」「最後に確定したページ番号」「ハッシュ履歴」を逐次保存。
- 再実行時は未完了の段・ページから続行。

## ⚙️ 5. 設定・ディレクトリ構成・運用手順

### 5.1 設定項目（`config.yaml`）

```yaml
book_title: "sample-book"
capture:
  region: [x, y, width, height]   # 読書領域（screencapture -R）。実測で決定
  page_turn_key: "right"          # right / left（右綴じ本は要確認）
  page_turn_wait: 1.0             # 秒
  stable_frames: 2                # 撮影確定に必要な連続安定フレーム
  end_detect_repeats: 3           # 同一ハッシュ連続で終了
  max_pages: 2000                 # 安全上限
preprocess:
  trim_margin: true
  target_dpi: 300
ocr:
  languages: ["ja-JP", "en-US"]
  recognition_level: "accurate"
build:
  image_format: "jpeg"            # or png。容量とのトレードオフ
  jpeg_quality: 85
```

### 5.2 ディレクトリ構成

```
kindle2pdf/
├─ config.yaml
├─ state.json
├─ src/
│  ├─ capture.py
│  ├─ preprocess.py
│  ├─ ocr.py
│  ├─ build_pdf.py
│  └─ pipeline.py
└─ work/<book_title>/
   ├─ raw/       # 撮影生画像 page_0001.png ...
   ├─ pages/     # 前処理後の確定ページ
   ├─ ocr/       # page_0001.json（text/bbox/confidence）
   └─ output/    # <book_title>.pdf
```

### 5.3 事前準備（一度だけ）

1. **Kindle for Mac** をインストールし対象書籍をダウンロード。
2. **表示設定**: 単一ページ表示、フォントサイズ固定、余白/行間固定、（可能なら）ダーク非適用。
3. **macOS権限**:
   - スクリプト実行元（Terminal/Python）に **アクセシビリティ** 権限（キー送出のため）。
   - 必要に応じて **画面収録** 権限（`screencapture`の対象取得のため）。
4. **依存導入**: `python3`, `pip install ocrmac img2pdf reportlab pillow imagehash pyyaml`, （代替用）`brew install cliclick`。
5. **撮影領域 `region` のキャリブレーション**: Kindleウィンドウを固定位置に置き、読書領域だけを囲む座標を実測。

### 5.4 実行手順（想定）

```bash
# 1. Kindleで対象書籍を1ページ目で開いておく
# 2. パイプライン実行（全自動）
python src/pipeline.py --config config.yaml --book "sample-book"
# → capture → preprocess → ocr → build を順次実行
# 中断後の再開も同じコマンドでレジューム
```

## 🛠 6. Claude Code向け 実装計画

### 6.1 実装順序（段階的に検証）

1. **PoC-1: 撮影領域キャリブレーション** — `screencapture -R`で読書領域だけを1枚撮り、目視確認。
2. **PoC-2: ページ送り＋連続撮影** — 右矢印送出→待機→撮影のループを数十ページで検証。pHashログ出力。
3. **PoC-3: 最終ページ検出** — 実際の巻末で停止するか検証。閾値`end_detect_repeats`調整。
4. **PoC-4: OCR単体** — `ocrmac`で1ページOCR、日本語精度とbbox座標を確認。
5. **PoC-5: 透明テキスト層PDF** — 1ページ分の画像＋不可視テキストを重ねて検索可能を確認。
6. **統合: pipeline.py** — 4段連結＋`state.json`レジューム。
7. **仕上げ: config化・トリミング・圧縮・ログ整備**。

### 6.2 各段の検証（受け入れ基準）

- 撮影: 全ページが欠け・重複なく`pages/`に揃う。
- OCR: 本文テキストがPDFで**検索・選択**できる（サンプル語で検索一致）。
- PDF: ページ順・向き・サイズが正しく、ファイルが破損しない。
- レジューム: 途中Kill→再実行で続きから完了する。

### 6.3 想定リスク・未決事項（実装時に詰める）

- **`region`実測値**は環境ごとに異なる → キャリブレーション用の補助スクリプトを用意。
- **右綴じ/左綴じ**（コミック）でページ送りキー方向が逆 → 設定で切替。
- **見開き表示・章境界の半端ページ** → 単一ページ表示を強制、重複除去で吸収。
- **Vision→PDF透明層の座標変換**（Vision正規化座標→PDFポイント）の実装精度 → PoC-5で要検証。
- **画面スリープ/通知ポップアップ**が撮影に混入 → 実行中はスリープ抑止（`caffeinate`）、通知オフ推奨。

## 🧪 7. 実機PoC検証結果（2026-07-09 / 実書籍「はじめてのUXリサーチ」で確認）

MacBook Airの実機スクショ4枚（2940×1912, Retina）を用いて、机上設計を実データで検証した。**結果、設計を上書きすべき重要な発見が3件**あった。

### 7.1 【最重要】表示が「見開き2ページ（2カラム）」だった

1枚のスクショに左ページ＋右ページが写り、読み順は「左カラムを読み切って右カラム」。**全面OCR（Tesseract psm3）は左右の行を1行に混ぜてしまい、読み順が破綻**することを実証。**画像を中央で左右分割してから各カラムを個別OCR**すれば正しい読み順になることを確認した。
→ **設計変更**: 「Kindleを単一ページ表示に強制」または「見開き画像を左右2分割してページ化」を**必須工程**とする。本PoCは後者（4枚→8ページ）で成功。

### 7.2 macOSスクショのフローティングサムネイル写り込み

`⌘⇧4`撮影後に出る右下サムネイルが後続撮影に写り込んでいた。pHashで検証すると、**同一ページでもサムネ有無だけで距離6の差**が出る（下表）。最終ページ検出を誤らせる致命的要因。
→ **設計確定**: 本番撮影は**`screencapture` CLI（サムネイルを出さない）**を採用。これにより「同一＝距離0」が保証され、閾値設計が安定する。

### 7.3 pHash閾値の実測

| 比較 | ハミング距離 | 意味 |
|------|------------|------|
| 連続する別ページ間 | 16〜26 | ページは十分に判別可能 |
| 完全同一画像 | 0 | 送り失敗/最終ページ |
| 同一ページ・サムネ有無のみ差 | 6 | サムネ除去が前提条件 |

→ **サムネイルを除去した前提で `距離 <= 2 を「同一ページ」**と判定すれば、最終ページ検出・送り失敗検出が安全に成立する見込み。

### 7.4 OCR精度（Tesseract jpn 実測）

検索可能PDF（8ページ）の生成に成功。テキスト層は有効で、単語検索は概ね機能した。ただし**日本語特有の弱点**を確認：
- 文字間に空白/改行が挿入されやすく、**語句検索が分断される**（例:「デプスインタビュー」は空白除去後のみヒット）。
- 一部語句の誤認識（「まず」→「ますず」、「一対複数」→「一対復数」、「半構造化インタビュー」「5W3H」は抽出不可）。

→ **設計判断を強化**: 本命OCRは **Apple Vision** を第一候補として維持（日本語精度・余分な空白の少なさで優位が見込まれる）。Tesseractは「動くが日本語の語句検索用途には物足りない」フォールバック位置づけが妥当。**Visionの実精度は次段で要実測**（下記）。

### 7.5 Apple Vision 実精度を実測（Tesseractと同一画像で比較）→ **Vision採用で確定**

Homebrew Python 3.12のvenvで`ocrmac`を実行し、同一画像でTesseractと比較した。

| 指標 | Apple Vision | Tesseract(jpn) |
|------|-------------|----------------|
| 文字間の余分な空白 | **0.01 / 字**（ほぼ無し） | **0.53 / 字**（半数の字に混入） |
| 認識文字数（かな漢字） | 979 | 883 |
| 語句「半構造化インタビュー」「5W3H」等の原文ヒット | ✅ | 生はヒットするが**PDF化で空白が焼付き検索破綻** |
| 見開き2カラムの読み順 | **自前でほぼ正しく処理** | 左右分割が必須 |

- **決定要因**: Tesseractの「文字間スペース挿入癖(0.53/字)」がPDFテキスト層に焼き込まれ、日本語の**語句検索を壊す**（7.4で実証）。Visionはこれがほぼ皆無で、検索可能PDFの実用度が段違い。
- **副次的発見**: Visionは見開き2カラムの読み順も自前で正しく並べた。7.1の左右分割工程を簡略化できる可能性（確実を期すならVisionのbbox座標で列認識ソートを実装）。
- **環境メモ**: システムPython 3.9はコンパイラ非搭載でビルド失敗、Homebrew Python＋venvで解決（PEP 668回避のためvenv必須）。

→ **本命OCRは Apple Vision（`ocrmac`）で確定。** Tesseractは非macOS環境向けフォールバックに降格。
